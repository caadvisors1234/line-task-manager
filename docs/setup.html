着手前の確認　実装プラン §0.3 の発注者確認事項7件(「対象外」ステータス追加可否・非表示管理列の追加可否・Dropboxフォルダ構成・社内通知グループの人数・既存タスクシートの移行方法・サマリへの中間状態表示・GAS実行アカウント)が解決済みであることを確認してから構築を開始します。

必要なアカウント・取得物

LINE公式アカウント保有済み
LINE Developersコンソールへのアクセス権。試験運用開始時にライトプラン(月5,000円 税別・5,000通/月)へ変更します(日次サマリが無料枠200通/月を超過するため、パイロット段階から必要)。

03構築ステップ

STEP 1〜5 を順に実施します。いずれも実装プラン §8(セットアップ手順)・§6.2(Dropbox認可フロー)の要約です。

STEP1
LINEチャネル設定
LINE Developersコンソール / LINE Official Account Manager

対象チャネル(Messaging API)の [Messaging API設定] タブで以下を設定します。

設定項目設定値備考
長期チャネルアクセストークン 発行 スクリプトプロパティ LINE_CHANNEL_ACCESS_TOKEN へ登録
BotのユーザーID 控える U で始まる値。LINE_BOT_USER_ID へ登録
グループトーク・複数人トークへの参加 ON デフォルトはOFF。グループでの受信に必須
Webhookの利用 ON
Webhookの再送 OFF 必ずOFF。GASは302を返すため全イベントが配信失敗として記録され、ONにすると正常処理済みのイベントまで重複再送されます
応答メッセージ・あいさつメッセージ OFF 二重応答・自動発言の防止
チャット(手動返信) 現状のまま Webhookとチャットは併用可能
料金プラン ライトプラン 月5,000円(税別)・5,000通/月。試験運用開始時に変更
STEP2
Dropboxアプリ作成とリフレッシュトークン取得
Dropbox App Console / GASエディタ
App Consoleでアプリを作成します。アクセスタイプは Scoped access、権限(スコープ)は files.content.write / sharing.write / sharing.read を付与します。APP_KEY / APP_SECRET を控えます。
次の認可URLをブラウザで開いて認可し、表示される認可コードを控えます。 
https://www.dropbox.com/oauth2/authorize?client_id=<APP_KEY>&response_type=code&token_access_type=offline
token_access_type=offline は必須です(これがないとリフレッシュトークンが発行されません)。
GASエディタから exchangeDropboxAuthCode('<認可コード>') を1回だけ実行し、リフレッシュトークンを取得してスクリプトプロパティ DROPBOX_REFRESH_TOKEN へ保存します。以後のアクセストークンは実行時に自動再発行されます(有効期間約4時間・キャッシュ運用)。
手順2〜3は連続して実施します(認可コードは短時間で失効)。GASプロジェクトが未作成の場合は、先に STEP 4 の手順1(プロジェクト作成・コード反映)まで済ませてから戻ってください。
STEP3
スプレッドシート初期構築
Googleスプレッドシート / GASエディタ
新規スプレッドシートを作成し、IDをスクリプトプロパティ SPREADSHEET_ID へ登録します。
GASエディタから setupSpreadsheet() を実行します。使い方シート(先頭タブ)＋5シート(タスク一覧・メッセージログ・顧客マスタ・設定・返信テンプレート)＋エラーログシート、ヘッダー(人が記入する列=緑/AIの列=墨色の色分け・説明メモ付き)、タスク状況10区分のプルダウン、行色分けの条件付き書式、L〜S列の非表示化が自動生成されます。
設定シートに一次受け定型文を登録し、返信テンプレートシートに既存のやり取りから初期5〜10パターンを登録します。自社メンバーuserIDリストは手動登録不要です(状態「社内」のグループで発言した人が自動で追記されます。05 疎通確認の項番4)。
共有設定を社内メンバーのみに限定します(リンク共有オフ)。
STEP4
GASデプロイとWebhook接続
GASエディタ / LINE Developersコンソール
GASプロジェクトを作成し(または clasp push で反映)、スクリプトプロパティを全件登録します(04節の一覧)。プロジェクトの所有は会社共用アカウントを推奨します。appsscript.json は timeZone: "Asia/Tokyo"・V8ランタイムとします。
checkConfiguration() を実行し、設定漏れゼロを確認します。
「デプロイ」→「新しいデプロイ」→ 種類: ウェブアプリ / アクセス: 全員(匿名を含む) / 実行ユーザー: 自分。デプロイIDを控えます。以後のコード更新は同じデプロイIDへの新バージョン発行で行います(毎回「新しいデプロイ」を作るとURLが変わり、LINE側の再設定が必要になるため)。
WebアプリURLの末尾に ?token=<WEBHOOK_VERIFY_TOKEN> を付けた値を、LINE DevelopersコンソールのWebhook URLに設定します。
Webhook URL設定後の「検証」ボタンは失敗しますが正常です(GASは302を返す仕様)。疎通確認は実メッセージで行います(05節)。
STEP5
トリガー設置
GASエディタ

installTriggers() を実行し、次の2つの時間主導型トリガーを設置します。

関数頻度備考
runAnalysisBatch 5分おき 1回の処理グループ数上限(初期値5)と4.5分の自己中断ガードあり
sendDailySummary 毎日 10〜11時枠 発火は枠内の任意時刻(一度決まると日々ほぼ固定)。案内表記は「10:00頃」

LINEコンソールの「検証」ボタン 失敗するが正常(GASは302を返す仕様のため)。以降の確認は実メッセージで行う
社内テストグループにBotを招待 顧客マスタに行が自動追加され、サロン名にLINEのグループ名が自動記入される
テストグループでテキスト送信 メッセージログに1行追加。発言者区分(自社/お客様)が正しい

5分待つ(またはバッチ手動実行) タスク一覧に起票され、タスク状況「未対応」・行の色が正しい

08構築時の留意事項

実装プラン §7(技術的制約とリスク対策17項目)から、構築時に特に踏みやすいものを抜粋します。

「検証」ボタンの失敗は仕様 GASのWebアプリは302を返すため、LINEコンソールのWebhook検証は失敗しますが、doPost自体は正常に実行されます。疎通確認は必ず実メッセージで行います。
Webhook再送は必ずOFF 302応答によりLINE側では全イベントが「配信失敗」として記録されます。再送をONにすると正常処理済みのイベントまで繰り返し再送され、処理量が数倍になります。再送に頼らない前提で、doPost内は3回リトライ＋重複排除で防御します。
署名検証は構造的に不可 GASのdoPostはHTTPリクエストヘッダーを取得できないため、LINE公式推奨の署名検証(X-Line-Signature)は実装できません。代替としてURLの秘密トークン＋destination 照合の二段防御を行います(トークンは年1回ローテーション)。
日次サマリは「10:00頃」 GASの日タイマートリガーは指定した1時間枠内の任意時刻に発火するため、「10:00ちょうど」は保証できません。社内への案内も「10:00頃」に統一します。
画像・ファイルは受信時に即保存 LINEのコンテンツ取得(api-data.line.me — 通常APIとドメインが異なる)は保持期間経過後に失敗するため、分析バッチに回さずWebhook受信時に同期取得してDropboxへ保存します。50MB超はスキップし管理者へ通知します。
1グループに公式アカウントは1つまで 他社のBotが既に参加しているグループには追加できません。該当時は新グループを作成して移行します。
通数はグループ人数分カウント 日次サマリ1通でも受信人数分(約30名)の通数を消費します。ライトプランは追加購入不可のため、送信前に通数残量をAPIで確認し、しきい値(4,500通)超過で管理者へ警告します。
