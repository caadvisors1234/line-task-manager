## 1. 前提と全体アーキテクチャ

### 1.1 前提条件（本書で変更しない決定事項）

| 項目 | 内容 |
|---|---|
| メッセージ量 | 20〜100件/日（全顧客合計） |
| Bot の振る舞い | グループ内で一切発言しない（受信・記録のみ）。お客様への返信・一次受け定型文の送信はすべて担当者が行う |
| LINE | 保有済みLINE公式アカウント（Messaging API）。通知配信のためライトプラン（月5,000円・税別、5,000通/月）を試験運用開始時から利用 |
| AI | Gemini API、モデル `gemini-3.1-flash-lite`（APIキー取得済み）。**有料ティア**で利用し、入力を学習に使わせない設定とする |
| 処理基盤 | Google Apps Script（GAS）。サーバー等の追加機材なし |
| データ保存 | Googleスプレッドシート1ファイル（タスク一覧・メッセージログ・顧客マスタ・設定・返信テンプレート） |
| 受領データ | 画像・ファイルは受信時に即Dropboxへ自動保存し、共有リンクを「議事録・添付資料」列に記録 |
| 通知 | 毎朝9時台に社内LINEグループへ日次サマリをPush（1日1通） |
| 移行 | パイロット5〜10店舗 → 段階拡大 |

### 1.2 コンポーネント構成

```
┌─ お客様グループ(サロンごと) ─────────────────────────┐
│ お客様 + 会社LINE(個人アカウント) + 記録用アカウント(公式アカウント) │
└───────────────┬─────────────────────────┘
                │ Webhook(受信は課金対象外)
                ▼
┌─ Google Apps Script ─────────────────────────────┐
│ [系統1] doPost: 受信 → メッセージログ保存                  │
│         画像・ファイルは即時取得 → Dropbox保存             │
│ [系統2] 分析バッチ(5分おき): 未分析分をGeminiで分析         │
│         → タスク一覧へ起票・関連タスク紐付け・返信提案        │
│ [系統3] 日次サマリ(毎朝): 未対応・急ぎを集計                │
│         → 社内LINEグループへPush                        │
└───────┬───────────────────┬─────────────┘
        ▼                       ▼
┌─ スプレッドシート ─┐      ┌─ 社内LINEグループ ─┐
│ タスク一覧ほか各シート │      │ 毎朝のサマリ通知      │
└─────────────┘      └───────────────┘
        ▲
        │ 担当者: タスク状況更新・担当者名等の手動記入(Botは上書きしない)
```

### 1.3 処理フロー3系統

| 系統 | 起動 | 処理 | 遅延要件 |
|---|---|---|---|
| 1. Webhook受信 | LINEからのHTTP POST（Webアプリ） | イベント検証 → 重複排除 → メッセージログ追記 → 画像・ファイルは同期でDropbox保存 | 即時（コンテンツ失効前の確実な取得が目的） |
| 2. 分析バッチ | 時間主導型トリガー（5分おき） | 未分析メッセージをグループ単位でまとめ、直近会話＋未完了タスク＋返信テンプレートを文脈としてGeminiへ → タスク起票・紐付け・返信提案記入 | 5〜10分以内の起票で十分（日次サマリが主な共有手段のため） |
| 3. 日次サマリ | 時間主導型トリガー（毎朝9〜10時枠） | タスク一覧を集計 → サマリ文生成 → 社内グループへPush | 9:00頃（時間主導型トリガーは指定枠内の任意時刻に発火するため「9:00頃」と案内） |

## 2. GASプロジェクト構成

### 2.1 ファイル構成と責務

`src/` 配下に以下の .gs ファイルを配置する。GASはフラットな名前空間のため、ファイル外から呼ばない関数は末尾 `_` で統一する（`_` 付き関数はGASエディタの実行対象からも除外される）。

| ファイル | 責務 | 主要関数 |
|---|---|---|
| `main.gs` | Webhookエントリポイント。イベント振り分けのみ | `doPost(e)` / `handleEvent_(event)` / `handleMessageEvent_(event)` / `handleJoinEvent_(event)` |
| `config.gs` | 定数集約: シート名、列番号、タスク状況の値、メッセージ種別enum、スクリプトプロパティキー名 | `CONFIG` / `SHEET` / `COL` / `STATUS` / `MSG_TYPE`（定数オブジェクトのみ） |
| `lineClient.gs` | LINE Messaging APIラッパ | `pushMessage_(to, messages)` / `fetchMessageContent_(messageId)`（api-data.line.me） / `fetchGroupMemberProfile_(groupId, userId)` / `fetchGroupSummary_(groupId)` / `fetchQuotaConsumption_()` |
| `dropboxClient.gs` | Dropbox APIラッパ（OAuth2リフレッシュ含む） | `getDropboxAccessToken_()` / `uploadToDropbox_(blob, path)` / `getOrCreateSharedLink_(path)` / `exchangeDropboxAuthCode(authCode)`（セットアップ時のみ手動実行） |
| `geminiClient.gs` | Gemini API呼び出し（structured output） | `callGemini_(systemPrompt, userContent, responseSchema)` / `buildResponseSchema_(openTaskIds)` |
| `analyzer.gs` | 分析バッチ本体 | `runAnalysisBatch()`（5分トリガー） / `buildAnalysisContext_(groupId, targetMessages)` / `applyAnalysisResult_(msgRow, result)` |
| `messageLogRepo.gs` | メッセージログシートの読み書き | `appendMessageLog_(record)` / `getUnanalyzedMessages_(limit)` / `getRecentConversation_(groupId, count)` / `markAnalyzed_(rowIndexes, status)` / `isDuplicateEvent_(webhookEventId, messageId)` |
| `taskRepo.gs` | タスク一覧シートの読み書き（Botが書ける列を限定） | `createTask_(task)` / `issueTaskId_()` / `getOpenTasksBySalon_(salonName)` / `getTasksForSummary_()` / `appendAttachmentLink_(taskId, url)` |
| `masterRepo.gs` | 顧客マスタ・設定・返信テンプレートの読み取り | `resolveSalonName_(groupId)` / `registerNewGroup_(groupId)` / `getInternalUserIds_()` / `getSettings_()` / `getReplyTemplates_()` |
| `notifier.gs` | 日次サマリ・管理者通知 | `sendDailySummary()`（日次トリガー） / `summarizeTasks_(tasks, options)` / `buildSummaryFlex_(tasks, options)` / `buildSummaryText_(tasks, options)`（フォールバック用） / `notifyAdmin_(message)` |
| `setup.gs` | 初期構築ワンタイム関数 | `setupSpreadsheet()`（シート・プルダウン・条件付き書式・使い方シート・ヘッダー色分けとメモの生成。§3.7） / `installTriggers()` / `checkConfiguration()`（設定漏れ検査） |
| `utils.gs` | 共通処理 | `withScriptLock_(fn, timeoutMs)` / `fetchWithRetry_(url, params, maxRetry)` / `formatDateTime_(date)` / `logError_(context, error)` |
| `test.gs` | 手動テスト用 | `test_simulateTextMessage()` / `test_simulateImageMessage()` / `test_runAnalysisOnFixture()` / `test_buildSummary()` / `test_buildSummaryFlex()` |

### 2.2 スクリプトプロパティ一覧（秘匿情報）

| キー | 用途 | 取得元 |
|---|---|---|
| `LINE_CHANNEL_ACCESS_TOKEN` | Push送信・コンテンツ取得・プロフィール取得 | LINE Developersコンソール（長期チャネルアクセストークン） |
| `WEBHOOK_VERIFY_TOKEN` | Webhook URLの秘密トークン（`?token=` 照合用） | 実装者が乱数生成（32文字以上） |
| `LINE_BOT_USER_ID` | Webhookボディの `destination` 照合用 | LINE Developersコンソール（BotのユーザーID、`U` で始まる値） |
| `GEMINI_API_KEY` | Gemini API認証 | Google AI Studio（取得済み。有料ティアのプロジェクトに紐づくキーであること） |
| `DROPBOX_APP_KEY` / `DROPBOX_APP_SECRET` | Dropbox OAuth2 | Dropbox App Console |
| `DROPBOX_REFRESH_TOKEN` | アクセストークン再発行 | 初回認可フロー（§8.2）で取得 |
| `SPREADSHEET_ID` | 対象スプレッドシート | 作成したシートのURL |
| `SUMMARY_GROUP_ID` | 日次サマリの宛先（社内グループ） | BotをグループにいれてWebhookログから取得（§8.5） |
| `ADMIN_GROUP_ID` | エラー・警告通知の宛先 | 同上（少人数の管理者グループを推奨。通数節約のため） |

シート「設定」には運用者が調整してよい非秘匿値のみを置く（§4.5）。秘匿値は必ずスクリプトプロパティに置き、シートには書かない。

### 2.3 デプロイ構成

- `appsscript.json`: `timeZone: "Asia/Tokyo"`、`webapp: { access: "ANYONE_ANONYMOUS", executeAs: "USER_DEPLOYING" }`、`runtimeVersion: "V8"`
- Webアプリとしてデプロイし、URL末尾に `?token=<WEBHOOK_VERIFY_TOKEN>` を付けた値をLINEのWebhook URLに設定する。
- **デプロイIDを固定して「バージョンを管理」から新バージョンを発行**する運用とする（毎回「新しいデプロイ」を作るとURLが変わり、LINE側の再設定が必要になるため）。

### 2.4 トリガー構成（`installTriggers()` が設置）

| 関数 | 種別 | 頻度 | 備考 |
|---|---|---|---|
| `runAnalysisBatch` | 時間主導型 | 5分おき | 1回の処理件数上限と自己中断ガードあり（§4.3） |
| `sendDailySummary` | 時間主導型（日タイマー） | 毎日 9〜10時枠 | 発火時刻は枠内の任意時刻（一度決まると日々ほぼ固定）。正本の「毎朝9時台・9:00頃」に合わせ9〜10時枠とする |

### 2.5 開発環境（clasp）

- リポジトリ直下に `.clasp.json`（`rootDir: "src"`）、`src/appsscript.json` を置き、`clasp push` / `clasp deploy --deploymentId <固定ID>` で反映する。
- 第1段階は素のJavaScript（.gs）とし、TypeScript化はしない（ビルド工程を持ち込まず、GASエディタ上での緊急修正を可能にするため）。
- `.gitignore` に `.clasprc.json`（認証情報）を追加する。

## 3. スプレッドシート詳細設計

1ファイルに5シート＋エラーログシート＋アーカイブ用シート、および説明専用の「使い方」シート（§3.7）を持つ。シート名・列は `config.gs` の定数と一致させ、`setupSpreadsheet()` が自動生成する。

### 3.1 シート1: タスク一覧

承認版提案書の「現行シートの列構成を踏襲＋メッセージ種別＋返信提案」に従い、以下の通り列番号を確定する。A〜J列が表示列（現行運用と同じ見た目）、K列以降は**非表示の管理列**（§0.3-2 の確認事項）。

| 列 | 項目 | 記入者 | 内容 |
|---|---|---|---|
| A | 対応期日 | AI(初期値のみ)→担当者 | メッセージから読み取れた場合にAIが初期値を記入。以後の確定・変更は担当者が行い、Botは再更新しない |
| B | 納品データ | 担当者 | 当社が制作・納品したデータのリンク等。**Botは一切書き込まない** |
| C | 担当者名 | 担当者 | **Botは一切書き込まない** |
| D | 店舗名 | AI | 顧客マスタから自動引き当て（例: サロンA様） |
| E | メッセージ種別 | AI | `新規依頼` / `回答・承認` / `質問・確認` / `資料送付` / `雑談・お礼`（雑談・お礼は原則起票しない） |
| F | 作業内容 | AI | 1行サマリ（例: ホットペッパー広告バナーの差し替え） |
| G | 議事録・添付資料 | AI(追記)＋担当者 | 受領データのDropbox共有リンクをAIが**追記**（既存セル内容には改行して追加し、上書きしない）。担当者も議事録リンク等を自由に記入 |
| H | タスク状況(進捗) | AI(初期値のみ)→担当者 | 起票時に「未対応」（進行承認の場合は「反映待ち」）を設定。以後の更新は担当者のみ |
| I | タスク発生日 | AI | 例: 「7/2 LINE」 |
| J | 返信提案 | AI | 返信の下書き。すぐに判断できない新規依頼には一次受け定型文を、それ以外は過去のやり取り・テンプレートに基づく返信案を記入（送信は担当者） |
| K〜R | （非表示管理列） | AI | K:タスクID(自動採番)（T-0001形式） / L:LINEグループID(システム用) / M:緊急度(AI判定)（高・中・低） / N:関連タスクID / O:要確認(AIの自信が低い印) / P:登録元メッセージID(システム用) / Q:タスク登録日時 / R:期限(システム用)（AI抽出の正規化日付 yyyy-MM-dd。日次サマリの「期限間近」判定に使用） |

**書き込み保護の実装:** `taskRepo.gs` は「Botが書き込める列」を定数 `COL.TASK.AI_WRITABLE = [A(初期値のみ), D, E, F, G(追記のみ), H(初期値のみ), I, J, K〜R]` として持ち、B・C列および既存行のA・H列への書き込みをコードレベルで禁止する。行の更新は「G列への追記」「J列の返信提案の追記」のみ許可し、それ以外の既存行更新は行わない。

**タスク状況のプルダウン（データ検証）と条件付き書式:** 現在の9区分をそのまま再現し、値に応じて行全体を自動で色分けする。

| 値 | 意味 | 行の色（承認版提案書モックの色コード。構築時に既存シートの実色と照合して確定） |
|---|---|---|
| 未対応 | 着手前の案件 | `#ffffff` 白（初期値） |
| 依頼中 | 対応を依頼し、返答・作業待ち | `#f4cccc` 薄赤 |
| 作業完了・未チェック | 作業完了、社内チェック前 | `#f4921e` 橙 |
| チェック完了・残りお客様連絡 | チェック済み。あとはお客様へ連絡 | `#ec47dd` マゼンタ（白文字） |
| タスク完了済み | すべて完了・クローズ | `#b7b7b7` 灰 |
| 佐藤さん提出 | 佐藤さんからの提出・確認待ち | `#ffe14d` 黄 |
| 急ぎの対応 | 最優先。最左列（対応期日）に赤字で期限を記入 | `#ff2b22` 赤（白文字） |
| 反映待ち | 下書き登録後、連絡まで完了したもの | `#4a86d8` 青（白文字） |
| お客様連絡待ち | お客様からの連絡待ち | `#b6d7a8` 薄緑 |
| （対象外） | §0.3-1 で追加が承認された場合のみ。AIの誤起票を人が除外する用途 | 灰＋取り消し線（新設のため任意） |

### 3.2 シート2: メッセージログ

全受信メッセージの生ログ。分析バッチの処理キューを兼ね、トラブル時の証跡・経緯確認に使う。

| 列 | 項目（シート上の列名） | 内容 |
|---|---|---|
| A | 受信日時 | JST、`yyyy-MM-dd HH:mm:ss` |
| B | LINEグループID(システム用) | LINEのgroupId |
| C | サロン名 | 顧客マスタから引き当て（未登録時は空欄） |
| D | 発言者の区分(自社／お客様) | `自社` / `お客様`（設定シートの自社メンバーuserIDリストで判定） |
| E | 発言者ID(システム用) | 発言者のuserId |
| F | 発言者の表示名 | グループメンバープロフィールAPIで取得。失敗時は「(取得不可)」 |
| G | メッセージの種類 | `text` / `image` / `file` / `video` / `audio` / `sticker` / `other` |
| H | メッセージ本文 | テキスト本文（非テキストはファイル名等のメタ情報） |
| I | メッセージID(システム用) | messageId。重複排除・コンテンツ取得に使用 |
| J | 受信イベントID(システム用) | webhookEventId。重複排除に使用 |
| K | 保存ファイルのリンク(Dropbox) | 非テキストの保存先共有リンク |
| L | AI分析の状態 | `未分析` / `分析済` / `分析対象外`（自社発言・スタンプ等） / `エラー` |
| M | AI分析の詳細データ(システム用) | Geminiの生出力JSON（デバッグ・プロンプト改善用） |
| N | 登録されたタスクID | このメッセージから起票・紐付けされたタスクID |
| O | AI分析の試行回数(システム用) | 分析バッチの失敗リトライ回数（5回で `エラー` へ。§4.3異常系） |

自社メンバーの発言も**必ず保存する**（進行承認の文脈判定に必須）が、分析ステータスは `分析対象外` とし、Geminiへの起票判定対象はお客様発言のみとする（自社発言は文脈としてのみ渡す）。

### 3.3 シート3: 顧客マスタ

| 列 | 項目 | 内容 |
|---|---|---|
| A | グループID | joinイベントで自動追加 |
| B | サロン名 | **Bot参加時にLINEのグループ名を自動記入**（グループ概要API `/v2/bot/group/{groupId}/summary`。取得失敗時は空欄）。表記の変更は人が上書きする（書き込みは新規登録時の1回のみでBotは再更新しないため、上書きが常に優先）。空欄の間はログのC列が空欄になり、日次サマリで「未設定グループあり」と警告 |

| D | Bot参加日 | 自動記録 |
| E | 備考 | 自由記入（将来: 既定担当者の登録欄として拡張） |

状態が `社内` のグループ（日次サマリ宛先・管理者グループ等）は、メッセージログ保存・分析・「サロン名未設定」警告の対象外とする（社内メンバーの雑談が誤起票されることを防ぐ）。

### 3.4 シート4: 設定

運用者が調整してよい非秘匿値のみ（秘匿値はスクリプトプロパティ。§2.2）。

| 列 | 項目 | 内容 |

| A | パターン名 | 例: 画像差し替え依頼への受領連絡 |
| B | 適用の目安 | 種別・キーワードの目安（AIがプロンプト内で参照する説明文） |

### 3.6 行数増加対策（アーカイブ方針）

- メッセージログ: 100件/日 ≒ 3.6万行/年。**月次で「ログ_yyyyMM」シートへ移動**する手動運用とし、パイロット後に必要ならトリガー化する。分析バッチの未分析検索は末尾500行のみを対象とし、全行走査をしない。
- タスク一覧: 「タスク完了済み」（および「対象外」）の行を四半期ごとにアーカイブシートへ移動する。
- スプレッドシート全体のセル上限（1,000万セル）に対し、上記運用で数年は問題にならない。

### 3.7 表示レイヤーの設計（使い方シート・ヘッダー色分け・セルメモ）

非エンジニアの担当者向けの表示改善。コードは `COL` の列番号で動作しヘッダー文字列に依存しないため、以下はすべて表示レイヤーのみの設計である（唯一の例外は設定シートA列の項目名で、`SETTING_KEY` と一致必須のため変更しない）。

- **使い方シート**: 先頭タブに説明専用シート「使い方」を置く（`buildGuideSheet_`。プログラムからは読み書きしない）。内容: 全体の流れ／各シートの役割／タスク状況の色付き凡例（`TASK_STATUS_STYLES` を再利用した静的塗り）／列の記入分担／よくある質問。
- **ヘッダーの色分け**: 全シートの見出し行を「人が記入・更新する列= `#d9f2e5`（緑）」「AIが自動記入する列= `#24292e`（墨色・白文字）」の2色で塗り分ける。タスク一覧では A・B・C・H が緑（A・H はAI初期値→人が更新のため人側に含める）。
- **セルメモ**: 各列見出しに「【記入する人】／【内容】／【例】」の3ブロック定型でメモ（setNote）を付与する。
- **列名の平易化**: カタカナ技術用語は日本語＋「(システム用)」付記へ（例: messageId→メッセージID(システム用)）。用語「起票」はシート上・通知上の表記では「登録」に統一する（本書の設計用語としての「起票」は維持）。
- ヘッダー定義（列名・メモ・色区分）は `setup.gs` の `*_HEADER_DEFS` 定数に集約する。変更の反映は、対象シートを削除して `setupSpreadsheet()` を再実行する（既存シートスキップの冪等設計のため。設定シートを再生成した場合は「自社メンバーuserIDリスト」「一次受け定型文」の再記入が必要）。

## 4. 機能別詳細設計

各機能を「入力 / 処理ステップ / 出力 / 異常系」で定義する。

### 4.1 Webhook受信（doPost）

**入力:** LINEプラットフォームからのHTTP POST（イベント配列）。

**処理ステップ:**

1. `e.parameter.token` を `WEBHOOK_VERIFY_TOKEN` と照合。不一致なら何もせず `OK` を返す（攻撃者に情報を与えない）。
2. ボディをパースし、`destination` を `LINE_BOT_USER_ID` と照合。不一致なら破棄。
3. 各イベントについて `webhookEventId` をCacheService（TTL 6時間）で照合し、既処理なら破棄。メッセージイベントは `messageId` でもメッセージログを照合する二重チェック（CacheServiceの保持期間は保証されず早期削除があり得るため、永続側の照合を必ず併用する）。
4. イベント種別で分岐:
   - `message`（グループ発のみ処理。1対1は対象外のため無視）:
     a. 顧客マスタからサロン名を引き当て（未登録グループならjoin漏れとして自動登録）。状態が `社内` のグループはここで処理を終了する（§3.3）。
     b. 発言者区分を判定（設定シートの自社メンバーuserIDリスト）、表示名を取得（失敗時「(取得不可)」で続行）。
     c. `image` / `file` / `video` / `audio` の場合、**この場で同期的に** `api-data.line.me` からコンテンツを取得しDropboxへ保存（§4.2）。テキストはこのステップをスキップし応答を早める。
     d. メッセージログへ1行追記（LockService保護下）。**お客様発言（テキスト・画像・ファイル・動画・音声）は `未分析`**、スタンプおよび自社発言は `分析対象外`（画像・ファイルも「資料送付」としての起票対象のため未分析に含める）。
   - `join`（Botがグループに追加された）: 顧客マスタへ自動追加（サロン名の初期値としてLINEのグループ名を取得・記入。§3.3）。
   - `leave`: 顧客マスタの状態を `退出` に更新。
   - その他（`memberJoined` 等）: 無視。
5. 常に `ContentService.createTextOutput('OK')` を返す。

**出力:** メッセージログの行、（メディア時）Dropbox保存とK列のリンク。

**Webhook再送の設計判断:** GASのWebアプリはPOSTに対して302リダイレクトを返すため、LINE側では全イベントが「配信失敗」として記録される。このため**Webhook再送はOFFとする**（ONにすると正常に処理済みのイベントまで繰り返し再送され、処理量が数倍になる）。再送に頼らない前提で、doPost内の各処理は失敗時に即時リトライし、それでも失敗した場合はエラーログと管理者通知で検知する。重複排除（手順3）は、設定変更や将来の仕様変更に備えた防御として維持する。

**異常系:**
- doPost全体をtry/catchで包み、**例外を外へ投げない**。エラーは `logError_` でエラーログシートに記録し、管理者通知（§4.6）。
- ロック取得失敗（30秒待機超過）やシート追記の失敗は、doPost内で最大3回再試行する（再送OFFのため、ここで取りこぼすと自動救済手段がない）。3回失敗時はエラーログ＋管理者通知とし、メッセージログとトーク履歴の突き合わせで手動復旧する。
- 二重登録の防御: 手順3の重複排除とDropboxパスの冪等性（§4.2）による。

### 4.2 画像・ファイルのDropbox即時保存

**入力:** messageId、メッセージタイプ、サロン名、（fileイベントの）ファイル名。

**処理ステップ:**

1. `GET https://api-data.line.me/v2/bot/message/{messageId}/content` でBlobを取得（通常APIの `api.line.me` とはドメインが異なる点に注意）。動画・音声は変換完了前だと取得できないため、`GET .../content/transcoding` で変換状態を確認し、`processing` の場合は2秒間隔で最大3回待って再取得する（それでも未完了ならmessageIdをログに記録し、分析バッチ側で遅延再取得する）。
2. アクセストークンを `getDropboxAccessToken_()` で取得。リフレッシュトークンから `POST https://api.dropboxapi.com/oauth2/token`（`grant_type=refresh_token`）で再発行し、CacheServiceに3.5時間キャッシュ（アクセストークンの有効期間約4時間より短く設定）。
3. `POST https://content.dropboxapi.com/2/files/upload` でアップロード。
   - 保存パス: `/LINEタスク管理/{サロン名}/{yyyyMM}/{yyyyMMdd_HHmmss}_{messageId}.{拡張子}`。日時は**Webhookイベントの `timestamp`**（重複配信でも不変）から生成し、messageIdと合わせて**同一メッセージは常に同一パス＝冪等**とする（`mode: overwrite`）。
   - サロン名が未設定のグループは `/LINEタスク管理/_未設定/{groupId}/` 配下に保存する。サロン名に含まれるパス不可文字（`/` `\` 等）は `_` に置換する。
   - `Dropbox-API-Arg` ヘッダーは非ASCII文字を受け付けないため、パスに日本語（サロン名）を含むJSONは **`\uXXXX` にエスケープして載せる**（HTTP header safe JSON）。
4. `POST https://api.dropboxapi.com/2/sharing/create_shared_link_with_settings` で共有リンクを作成。**409（`shared_link_already_exists`）の場合は、エラーレスポンスに含まれる既存リンクのメタデータを使うか、`sharing/list_shared_links`（`direct_only: true`）で既存リンクを取得**して使う。
5. リンクをメッセージログK列に記録。分析バッチでのタスク起票時にG列（議事録・添付資料）へ追記される。

**出力:** Dropbox上のファイル、共有リンク。

**異常系:**
- ファイルサイズがUrlFetchAppの上限（50MB）を超える場合: 保存をスキップし、ログのK列に「サイズ超過のため未保存」と記録して管理者通知。タスク起票は本文情報のみで続行。
- Dropbox認証エラー（リフレッシュトークン失効・アプリ削除等）: 管理者へ即時通知（受領データの取りこぼしに直結するため最重要アラート）。コンテンツ取得だけでも成功していれば、messageIdをログに残し、LINE側の保持期間内なら手動再保存の余地を残す。

### 4.3 分析バッチ（Gemini連携）

**入力:** メッセージログの `未分析` 行（末尾500行から検索）。

**処理ステップ:**

1. 開始時刻を記録。`未分析` 行をグループIDごとにまとめる（同一グループの連続メッセージを1回の分析にまとめ、API呼び出しを削減）。
2. 設定値「処理グループ数上限」（初期値5）まで、グループごとに:
   a. `buildAnalysisContext_` で文脈を構築: 直近会話ウィンドウ（設定値、初期10件。**自社発言を含む**、発言者区分付き）＋当該サロンの未完了タスク一覧（タスクID・作業内容・タスク状況）＋返信テンプレート＋一次受け定型文＋現在日時（JST）。
   b. `callGemini_` でstructured output呼び出し（§5.1・§5.3）。応答は対象メッセージごとの判定配列。
   c. 判定結果を検証: `relatedTaskId` が渡した未完了タスクIDの一覧に実在するかコード側でも照合（AIの幻覚対策の二重防御）。実在しない値は破棄し `要確認` に倒す。
   d. タスク化要と判定されたものは `createTask_` で起票（タスクID採番はLockService保護下）。既存タスクへの回答・承認の場合も**既存行は一切更新せず**、新規行として起票し、新規行のN列（関連タスクID）に元タスクのIDを設定する（既存行の手動記入を守るため）。
   e. メッセージログの分析ステータス・分析結果JSON・起票タスクIDを更新。
3. 経過時間が4.5分を超えたら残りを次回バッチへ持ち越して終了（GASの6分制限ガード）。

**出力:** タスク一覧の新規行、メッセージログの更新。

**異常系:**
- Gemini 429/5xx: 指数バックオフで2回まで再試行。それでも失敗した場合は `未分析` のまま残し（次回5分後に再試行）、メッセージログO列（分析試行回数）を加算して5回に達したら `エラー` にして管理者通知。
- 応答がスキーマ不一致・パース不能: 1回だけ再試行し、失敗なら `エラー`。**取りこぼし防止を最優先とするため、`エラー` 行は日次サマリに「分析失敗◯件（ログ確認）」として必ず表示する**。
- 起票の重複: `起票元messageId`（P列）で既存タスクを照合し、同一messageIdからの再起票を防ぐ。

### 4.4 日次サマリ通知

**入力:** タスク一覧の全未完了行（タスク状況が「タスク完了済み」「対象外」以外）。

**処理ステップ:**

1. 通数残量チェック: `GET https://api.line.me/v2/bot/message/quota/consumption` で当月消費を取得し、しきい値（設定値、初期4,500）超過なら管理者警告を付加。
2. 区分ごとに集計（承認版提案書の通知イメージに準拠）:
   - **急ぎ・期限間近**: タスク状況「急ぎの対応」、および期限が本日から3日以内（設定値）のもの。期限の判定はR列（AI抽出の正規化日付）で行う（A列の対応期日は「6/30〜7/1の間で更新」等の自由書式のため判定には使わない）。期限昇順。
   - **未対応・依頼中**: タスク状況「未対応」「依頼中」。
   - **お客様連絡待ち・反映待ち**: 件数のみ表示。
   - 中間状態（作業完了・未チェック/チェック完了・残りお客様連絡/佐藤さん提出）は承認版の通知イメージに合わせて表示しない（意図的な仕様。表示要否は§0.3-6の確認事項）。
3. Flexメッセージ（bubble）を生成（`buildSummaryFlex_`。絵文字なし。Flexでも通数課金はテキストと同一）:
   - 墨色（`#24292e`）のヘッダー帯にタイトル＋日付（例: 7/2(木) 9:00ごろ）。
   - 区分見出し＋件数。件数は1件以上でアクセント緑 `#06c755`、0件で灰。各タスクは「ラベル｜サロン名（太字）／作業内容／期限・※要確認」の縦組み。
   - 急ぎの強調は色を使わず「急ぎ｜」接頭辞＋太字のみ（配色を墨色＋アクセント1色に限定するため。赤は不使用）。
   - 0件の区分は「なし」と表示し、全区分0件でも同構造のbubbleを送る（毎朝同じ形が届くこと自体が稼働確認を兼ねる）。
   - フッターに「タスク一覧を開く」ボタン（URIアクションでスプレッドシートを開く。`style: primary` / 背景 `#06c755`）。URLには `openExternalBrowser=1` を付与し、LINE内ブラウザ（Google未ログインのためログイン画面に遷移してしまう）ではなく端末の標準ブラウザで開く。テキスト版の「詳細:」URLも同様。
   - 通知欄・トーク一覧に出る `altText` は「本日のタスクサマリ(7/2) 急ぎ・期限間近2件／未対応・依頼中3件」形式の短文（仕様上限1,500字に対し十分短い）。
   - 実装上の注意: Flexの `text` コンポーネントに空文字を渡すとHTTP 400になるため、空になり得る行はコンポーネント自体を生成しない。
4. 各区分を設定値（初期15件）で打ち切り「ほか◯件はシート参照」と付記。加えてFlex JSONが27KB（bubble上限30KBへの安全マージン）を超える場合はテキスト版に切り替えて送信する。
5. `SUMMARY_GROUP_ID` へPush送信（1通。グループ人数分の通数を消費する点に留意）。

送信失敗時フォールバック用のテキスト形式（`buildSummaryText_`。従来形式のまま維持し、`test_buildSummary` で検証。テキスト時は5,000文字上限に同じ切り詰めを適用）:

```
本日のタスクサマリ(7/2 9:00頃)
── 急ぎ・期限間近 2件 ──
[急ぎ] サロンB様|キャンペーン用バナーの当日修正(本日17:00)
[期限] 銀座整体院様|開始日変更の反映(6/30〜7/1)
── 未対応・依頼中 3件 ──
アース大槻様|インスタ投稿用画像の依頼(毎月希望)
サロンC様|掲載文の修正依頼
ガーデンウシワカマル様|看板画像の確認
── お客様連絡待ち 2件|反映待ち 1件 ──
詳細: https://docs.google.com/spreadsheets/d/...
```

**出力:** 社内LINEグループへの通知1通。

**異常系:** 3段フォールバック。(1) Flex送信失敗（JSON不正・5xxリトライ超過等）はエラーログへ記録の上、テキスト版で再送する。(2) テキスト版も失敗した場合（429=月次上限到達を含む）は管理者通知＋ログ記録。上限到達時はスプレッドシート閲覧を代替手段として案内する。なお「5xx応答だが実は配信済み」の稀なケースではFlexとテキストが二重に届き得る（Pushリトライと同水準のリスクとして許容）。

### 4.5 タスク起票・関連タスク紐付けの規則

- 起票対象: メッセージ種別が `新規依頼` / `質問・確認` / `資料送付`、および `回答・承認` のうち**次のアクションが生じるもの**（進行承認=反映作業が発生、等）。`雑談・お礼` は起票しない。
- 判定に迷う場合は**起票に倒す**（取りこぼしより誤検知を許容。人がタスク状況で除外する）。
- 初期タスク状況: 新規依頼・質問・資料送付=「未対応」、進行承認=「反映待ち」。
- 既存案件への回答・承認は関連タスクID（N列）で紐付け、確度が低い場合は要確認フラグ（O列）を立てる。要確認のタスクは日次サマリの該当行に「※要確認」を付記する。
- 資料送付のみのメッセージ（本文なしの画像等）は、直近の未完了タスクに関連付く可能性をAIが判定し、関連が高ければ既存タスクへの紐付け行として起票、独立した送付なら新規起票する。

### 4.6 エラーハンドリング・管理者通知

- すべてのcatch節は `logError_(context, error)` を通す。ログはメッセージログとは別の「エラーログ」シート（発生日時・関数・メッセージ・スタック）へ記録する。
- 管理者通知（`ADMIN_GROUP_ID` へのPush）は以下の条件のみ（通数節約と通知疲れ防止）:
  1. Dropbox認証エラー（最重要）
  2. 同一メッセージの分析5回失敗
  3. 日次サマリの送信失敗
  4. 通数しきい値超過
  5. サイズ超過等による保存スキップ
- 同一種別のエラー通知はCacheServiceで1時間に1回に抑制する。

## 6. セキュリティ・認証設計

### 6.1 Webhookの不正リクエスト対策

- **前提となる制約:** GASのWebアプリ（doPost）はHTTPリクエストヘッダーを取得できないため、LINE公式が推奨する署名検証（`X-Line-Signature` とチャネルシークレットによるHMAC-SHA256検証）は**構造的に実装できない**。
- 代替として二段の照合を行う:
  1. Webhook URLに付与した秘密トークン（`?token=<32文字以上の乱数>`）の一致。URLを知らない第三者のリクエストを排除。
  2. リクエストボディの `destination`（Botの ユーザーID）の一致。
- 上記で防げない「URL漏えい＋ボディ偽装」のリスクは、本システムが**送信・返信を一切行わない**（偽イベントが起きても誤ったタスク行が増えるだけで、外部への影響がない）ことから許容する。トークンは年1回ローテーションする（新デプロイ不要。プロパティとLINE側URLの更新のみ）。

### 6.2 Dropbox OAuth2（リフレッシュトークンフロー）

1. Dropbox App Consoleでアプリ作成（Scoped access / Full dropboxまたはApp folder。§0.3-3の決定に従う）。権限: `files.content.write` `sharing.write` `sharing.read`。
2. 認可URL: `https://www.dropbox.com/oauth2/authorize?client_id=<APP_KEY>&response_type=code&token_access_type=offline`（**`token_access_type=offline` が必須**。これがないとリフレッシュトークンが発行されない）。
3. ブラウザで認可し、表示された認可コードを `exchangeDropboxAuthCode('<コード>')` の引数に貼ってGASエディタから1回実行 → `POST https://api.dropboxapi.com/oauth2/token`（`grant_type=authorization_code`）でリフレッシュトークンを取得し、スクリプトプロパティへ保存。
4. 以後は `getDropboxAccessToken_()` が `grant_type=refresh_token` で短命アクセストークンを再発行し、CacheServiceにキャッシュして使い回す。有効期間は現行実測で約4時間（`expires_in: 14400`）だが公式保証値ではないため、応答の `expires_in` から10%引いた秒数をキャッシュ期間とする実装にする。
5. リフレッシュトークンは明示的に無効化しない限り長期有効。失効時（§4.2異常系）は手順2〜3を再実行する。

### 6.3 秘匿情報の管理区分

| 区分 | 保管場所 | 例 |
|---|---|---|
| 秘匿値 | スクリプトプロパティ | 各種トークン・APIキー・シークレット（§2.2の全項目） |
| 運用調整値 | 設定シート | 会話ウィンドウ件数、しきい値、定型文（§3.4） |
| コード内定数 | config.gs | 列番号、ステータス値、シート名 |

- スプレッドシートの共有は社内メンバーのみ（リンク共有オフ）。
- GASプロジェクトの編集権限は実装者・管理者に限定。
- リポジトリ（GitHub）にトークン類を一切コミットしない（`.clasprc.json` を `.gitignore` へ）。

### 6.4 個人情報の取り扱い

- メッセージ本文はGoogle（GAS・スプレッドシート・Gemini API）に送信され、受領ファイルはDropboxに保存される。
- Geminiは**有料ティア**で利用し、入力内容がモデルの学習に使われない扱いとする（承認版提案書の決定事項）。
- お客様への説明は提案書記載の案内文（「管理用のアカウントを追加させていただきます」）で行い、社内規程に沿って運用する。

## 8. セットアップ手順

### 8.1 LINE Developers / LINE公式アカウント設定

1. LINE Developersコンソールで対象チャネル（Messaging API）を開く。
2. [Messaging API設定] タブ:

   - 応答メッセージを **OFF**（あいさつメッセージもOFF。二重応答・自動発言の防止）。
3. LINE Official Account Managerで、チャットは現状の運用のまま（Webhookとチャットは併用可能）。

### 8.3 スプレッドシート初期構築

1. 新規スプレッドシートを作成し、IDを `SPREADSHEET_ID` へ。
2. GASエディタから `setupSpreadsheet()` を実行 → 使い方シート（先頭タブ）＋5シート＋エラーログシート、ヘッダー（色分け・セルメモ付き。§3.7）、プルダウン（タスク状況10区分）、条件付き書式（行の色分け・「急ぎの対応」行の強調）、K〜R列の非表示化が自動生成される。

1. GASプロジェクト作成（またはclaspでpush）。プロジェクトの所有は**会社共用のGoogleアカウント**を推奨（属人化の回避。アカウント種別によりクォータが異なる。§0.3-7）。スクリプトプロパティ（§2.2）を全件登録。

3. 「デプロイ」→「新しいデプロイ」→ 種類: ウェブアプリ、アクセス: 全員（匿名を含む）、実行ユーザー: 自分。デプロイIDを控える（以後のコード更新はこのIDへの新バージョン発行で行う）。
4. Webアプリ URL＋`?token=<WEBHOOK_VERIFY_TOKEN>` をLINEのWebhook URLに設定。
5. `installTriggers()` を実行（5分バッチ・日次サマリ）。

### 9.1 レベル1: 単体確認（GASエディタから test.gs を手動実行）

- サマリ生成: フィクスチャのタスク群から§4.4のフォーマット通りに組み立つこと（Flex版=`test_buildSummaryFlex`: 構造・文言・altText・空文字textなし・JSONサイズ上限。テキスト版=`test_buildSummary`: フォールバック形式）。件数超過時の切り詰め。

## 11. 運用・保守

### 11.1 通数・クォータ監視

- **LINE通数試算:** 日次サマリ1通/日 × 社内グループ約30名 × 30日 ≒ **900通/月**（Pushはグループの人数分カウント）。管理者通知（少人数グループ宛）を加えても5,000通/月の2〜3割程度。
- ライトプランは**追加メッセージ購入不可**のため、月中の残量を日次サマリ送信前にAPIで確認し、しきい値（4,500通）超過で警告する。将来「即時通知」等を追加する場合は宛先の少人数化またはスタンダードプラン移行を再検討する（提案書§今後の拡張）。
- 社内グループの人数が大きく増えた場合は試算を更新する（§0.3-4）。
- Gemini: 429発生回数をエラーログで月次確認。有料ティアの請求額を月次で確認（試算: 100件/日 × 入力約4,000トークン・出力約400トークンで月約5ドル＝700円前後。コストが問題になる場合は単価約1/3の gemini-2.5-flash-lite への切り替えも選択肢）。
- GASクォータ: Apps Scriptダッシュボードでトリガー実行時間を試験運用中は週次確認。

### 11.2 定常メンテナンス

| 周期 | 作業 |
|---|---|
| 週次（試験運用中） | 分類精度の突き合わせ・プロンプト調整（§5.4）、返信テンプレート追記 |
| 月次 | メッセージログのアーカイブ、Gemini請求確認、通数実績確認 |
| 四半期 | タスク完了済み行のアーカイブ、Webhookトークンの点検 |
| 年次 | `WEBHOOK_VERIFY_TOKEN` ローテーション、チャネルアクセストークン再発行 |

### 11.3 グループ展開の運用（フェーズ2〜4）

1. お客様へ案内文を送付（提案書の文例）。
2. グループにBotを招待 → 顧客マスタに自動追加 → サロン名を記入（これだけで運用に乗る）。
3. 週10〜20グループずつ追加し、追加週はサマリと実トークの突き合わせを行う。
