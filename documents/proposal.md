## 1. 背景と目的

- 美容室の集客支援において、お客様との連絡は会社LINE(個人アカウント)で実施している。今後もこの運用は継続する。
- 会社アカウント1つを複数名で共用しており、お客様からの**作業依頼タスク**や**質問・相談**の抜け漏れが発生している。
- 依頼タスクは現状、特に管理されておらず、トーク履歴の見返し(記憶ベース)に依存している。
- 代表担当サロンは「一次受け定型文→代表へ報告」の人手フローがあり、ここでも漏れ・遅延のリスクがある。

**目的:** お客様・会社LINE・Botを含むグループを作成し、メッセージを自動収集→AI分析→タスク化→社内リマインドすることで、抜け漏れゼロと円滑な作業移行を実現する。

## 3. 対策の全体像

```
┌─ お客様グループ(サロンごとに1つ) ──────────────┐
│  お客様(数名) + 会社LINE(個人アカウント) + Bot(公式アカウント) │
└──────────────┬───────────────────┘
               │ ① メッセージをWebhookで受信(受信は課金対象外)
               ▼
┌─ GAS(Google Apps Script) ─────────────────┐
│ ② メッセージログを全件保存(発言者を自社/お客様で区別)。
│    受信した画像・ファイルは即Dropboxへ保存しリンクを記録
│ ③ 数分おきにGemini API(gemini-3.1-flash-lite)で未分析分を分析。
│    直近の会話の流れ＋未完了タスクを文脈に渡し、メッセージ種別判定・
│    進行承認の検知・緊急度・期限・要約・返信提案を生成
│ ④ タスクをタスク一覧へ自動登録(既存案件への回答は関連タスクへ紐付け)
│ ⑤ 毎朝、未対応タスクの日次サマリを社内LINEグループへPush通知
└──────────────┬───────────────────┘
               ▼
┌─ スプレッドシート(1ファイル集約) ─┐   ┌─ 社内LINEグループ ─┐
│ ・タスク一覧(全顧客横断)        │   │ 毎朝のサマリ通知       │
│ ・メッセージログ              │   │ 「未対応タスク◯件…」   │
│ ・顧客マスタ/設定            │   └─────────────┘
│ 担当者がステータスを「完了」に更新   │
└───────────────────┘
```

**ポイント**

- お客様への返信は従来どおり**人(会社LINE個人アカウント)が行う**。Botは黙って収集・分析・通知するだけなので、個人アカウントならではの温度感・関係性はそのまま維持できる。
- LINEの仕様上、**個人アカウントの1対1トークはBotから取得できない**。そのため「お客様+会社LINE+Botのグループ」への移行がこの対策の前提となる(背景の構想どおり)。

## 4. システム構成


| 要素      | 採用技術                              | 役割                                |
| ------- | --------------------------------- | --------------------------------- |
| メッセージ収集 | LINE Messaging API(保有済み公式アカウント)   | グループ内メッセージのWebhook受信、日次サマリのPush送信 |
| 処理基盤    | Google Apps Script(無料・サーバー不要)     | Webhook受付、分析バッチ、日次通知のトリガー実行       |
| AI分析    | Gemini API(gemini-3.1-flash-lite) | カテゴリ分類・緊急度・期限抽出・要約(JSON出力)        |
| データ保存   | Googleスプレッドシート(1ファイル)             | タスク一覧・メッセージログ・顧客マスタ               |
| 通知      | LINE公式アカウントのPushメッセージ             | 社内LINEグループへ毎朝1通                   |


### AI分析の単位 — 1通ごとではなく「会話の流れ」で読む(確認①②への対応)

抜け漏れの多くは、単語ではなく**文脈**を読まないと判定できません(例:「ありがとうございます」だけの返信が、実は「進行承認」=作業を進めてよいという合図であるケース)。そこで本Botは、メッセージを1通ずつ単独で見るのではなく、**そのグループの直近の会話ウィンドウ(自社側の発言を含む)＋当該サロンの未完了タスク**をまとめてGemini に渡して分析します。gemini-3.1-flash-lite は広い文脈窓を持つため、この方式でも低コストで動作します。

- **自社/お客様の判別:** Botは参加グループ内の**全メンバーの発言(自社の個人アカウントの送信を含む)**をWebhookで受信できます。設定シートに「自社メンバーのuserIDリスト」を登録し、発言者を「自社」「お客様」に区別してメッセージログへ保存します。これにより「自社がアクションを予告 → お客様が承認」という流れを認識できます。
- **進行承認の検知(確認①):** 直前に自社が「下書き登録させていただきます」等のアクション予告をしている文脈で、お客様の「ありがとうございます」を**進行承認**と判定し、次工程をタスク化します(単語の拾い上げではなく前後の文脈で判断)。
- **連続メッセージの自動判別(確認②):** 当該サロンの未完了タスクを文脈に含めることで、新着メッセージが「既存案件への回答・承認」か「新規のご要望」かをAIが自動判定します。回答・承認の場合は、どの未完了タスクへの応答かを推定し**関連タスクID**を付与します(現在は人が目視で行っている振り分けを自動化)。確度が低い場合は **`要確認`** を立て、人が最終判断します。

### AI分析の出力(分類設計 — お任せいただいた部分)

各メッセージ(または会話のまとまり)を以下のJSONで構造化します。

- **メッセージ種別:** `新規依頼/要望` / `既存案件への回答・承認(進行承認を含む)` / `質問・相談` / `資料・データ送付` / `雑談・お礼(アクション不要)`
- **カテゴリ:** `作業依頼` / `質問・相談` / `日程調整` / `資料・素材共有` / `確認・承認待ち` / `その他・雑談`
- **タスク化要否:** 会社側のアクションが必要か(true/false)。**短文のお礼でも、文脈上「進行承認」と読めるものはタスク化します**(単なる相づち・雑談は対象外)
- **緊急度:** `高`(即日対応・クレーム・公開中広告の不具合等) / `中`(数日以内) / `低`(期限なし・軽微)
- **期限:** 本文から読み取れる場合に抽出(「金曜までに」→ 具体日付に変換)
- **要約:** 1行の作業内容サマリ(例:「Instagram広告のクーポン画像を6/15までに差し替え」)
- **関連タスクID:** 既存案件への回答・承認の場合に紐付け(確度が低い場合は `要確認`)
- **依頼者:** グループ内の発言者表示名

抜け漏れ防止を最優先とするため、判定に迷う場合は**タスク化に倒す**(取りこぼしより誤検知を許容し、人がシート上で「対象外」にする)方針とします。

## 5. スプレッドシート設計(1ファイル集約)

### シート1: タスク一覧(メインの管理画面 — 既存タスクシートの上位互換)

現在お使いのタスクシート(連絡日・店舗名・作業内容・担当者・対応期日・納品データ・タスク状況・音声指示など)の要素をすべて引き継ぎ、そこに**AIの自動入力**を重ねた一元管理シートです。列は **「AI自動入力」** と **「手動追記」** の2領域に明確に分けます。**Botは手動追記列を一切上書きしません(再分析時も含む)** — 担当者名・対応状況・音声指示メモといった人の記入が保護されます(要望①②)。

**AI自動入力(Botが記入・更新)**

| 列          | 内容                                |
| ---------- | --------------------------------- |
| タスクID      | 自動採番(T-0001…)                     |
| タスク発生日・場所  | 連絡日と発生元(例:「6/12 LINE」)            |
| 店舗名        | 顧客マスタから自動引き当て                     |
| メッセージ種別    | 新規依頼/回答・承認/質問・相談/資料送付/雑談(確認①②の判定結果) |
| 作業内容       | AIによる1行サマリ                        |
| 緊急度        | 高/中/低(条件付き書式で色分け)                 |
| 期限         | AI抽出(本文から読み取れる場合。なければ空欄)          |
| 関連タスクID    | 既存案件への回答・承認時に紐付け(確度が低い場合は`要確認`)   |
| 返信提案       | 過去パターンに基づくAI生成の返信案(要望④)           |
| 納品データ      | 受領データのDropbox共有リンク(自動保存・要望③)      |
| 元メッセージ     | 原文(複数メッセージは連結)                    |

**手動追記(人が記入・Botは触れない)**

| 列        | 内容                            |
| -------- | ----------------------------- |
| 担当者名     | 対応する担当者(要望①)                  |
| 対応状況     | ステータス(下記プルダウン。行の色で進捗を管理)      |
| 対応指示     | 対応方針・指示の補足(要望②)               |
| 音声指示メモ   | 後日届く音声指示の書き起こし・メモ(要望②)        |
| 対応期日     | 運用上の確定期日(AI抽出の「期限」を参考に人が確定)   |
| 議事録・添付資料 | 関連する議事録・資料へのリンク等              |

**対応状況(ステータス)** は既存運用の色管理をそのまま踏襲し、プルダウン＋**条件付き書式で行を自動的に色分け**します:
`未対応` / `依頼中` / `作業完了・未チェック` / `チェック完了・残りお客様連絡` / `タスク完了済み` / `佐藤さん提出` / `反映待ち`(SALON BOARD下書き登録後〜連絡まで) / `お客様連絡待ち` / `対象外`。「急ぎの対応」は緊急度=高＋行頭を赤表示で踏襲します。

サロン名・対応状況・担当者でフィルタすれば「未対応のタスク」「あるサロンの依頼履歴」「自分の担当案件」が即座に見えます。担当交代時の引き継ぎ(円滑な作業移行)にもこのシートがそのまま使えます。

### シート2: メッセージログ

全受信メッセージの生ログ(日時・グループ・**発言者区分(自社/お客様)**・発言者名・本文・分析ステータス)。発言者区分は進行承認・回答の判定(確認①②)に用います。分析バッチの処理キューを兼ね、トラブル時の証跡にもなります。

### シート3: 顧客マスタ

LINEグループID ⇔ サロン名の対応表。新しいグループにBotを招待すると自動で行が追加され、サロン名を記入するだけで運用に乗ります。(将来、サロンごとの既定担当者を登録すれば、タスクの担当者自動アサイン・担当者別通知に拡張できます)

### シート4: 設定

社内通知グループID、通知時刻、Geminiプロンプトの調整項目に加え、以下を保持します。

- **自社メンバーのuserIDリスト:** 発言者の自社/お客様判別に使用(確認①②)
- **Dropbox認証情報:** アプリのclient id/secret＋長期有効なリフレッシュトークン、保存先フォルダ(要望③)。秘匿性の高い値はスクリプトプロパティ側で管理します

### シート5: 返信テンプレート

文面パターン ⇔ 返信例の対応表(要望④)。過去のやり取りから抽出した定型的な返信を登録しておき、AIが返信提案を生成する際の手本として参照します。運用しながら追記して精度を高めます。

## 6. 運用フロー(人の動き)

### 日常運用

1. お客様がグループでメッセージ送信 → 担当者は**従来どおり返信**(運用は何も変わらない)。返信時はシートの **「返信提案」列のAI生成案を下書きとして参考**にできます(要望④。送信はあくまで人が行います)
2. Botが裏で収集・分析(会話の流れを読んで種別判定・進行承認の検知)し、タスクをシートに自動登録。**受領した画像・ファイル・データは自動でDropboxに保存**され、リンクが「納品データ」列に記入されます(要望③)
3. 担当者は **手動追記列(担当者名・対応状況・対応指示・音声指示メモ等)を随時記入**。後日届く音声指示も、同じタスク行の「音声指示メモ」に追記して一元管理します(要望①②)
4. **毎朝(例: 9:00)** 社内LINEグループにサマリが届く:
  ```
   📋 本日のタスクサマリ(6/10 9:00)
   ── 未対応 5件 ──
   🔴[高] サロンA|広告画像の差し替え(期限 6/12)
   🟡[中] サロンB|クーポン内容の変更相談
   🟡[中] サロンC|掲載文の修正依頼
   ⚪[低] サロンD|次回打ち合わせ日程の相談
   ⚪[低] サロンE|料金プランについて質問
   ── 対応中 3件 / 昨日完了 4件 ──
   詳細: https://docs.google.com/spreadsheets/d/...
  ```
5. 対応した社員がシートの**対応状況を更新**(行の色で進捗が一目で分かり、翌朝のサマリにも反映されます)

### 代表担当サロンのフロー改善

連絡があった事実と内容要約が翌朝のサマリに**必ず**載るため、「定型文→口頭・個別報告」が漏れても代表が確実に把握できます。報告フロー自体は当面現状維持とし、Botを安全網として重ねる形です(定型文の自動送信は将来検討)。

## 8. コスト試算


| 項目                     | 費用                                             |
| ---------------------- | ---------------------------------------------- |
| GAS                    | 無料(無料枠で十分: URL Fetch 2万回/日 ≫ 100件/日)           |
| LINE Messaging API     | **ライトプラン 月5,000円(税別)・5,000通/月**(受信は課金対象外。Pushは受信者の人数分カウント: 日次サマリ30回/月 × 社内グループ約30名 ≒ **900通/月**) |
| Gemini API(flash-lite) | 無料枠内の見込み(無料枠の目安: 15リクエスト/分・1,000件/日 ≫ 100件/日。仮に有料でも月数十〜200円程度) |
| スプレッドシート               | 無料                                             |
| **合計**                 | **月5,000円(税別)**                                |


※LINEのPushメッセージは「送信先グループの人数分」課金カウントされます。社内通知グループが約30名のため、日次サマリだけで約900通/月となり無料枠(コミュニケーションプラン200通/月)を超過することから、**ライトプラン(月5,000円・5,000通/月)を採用**します。日次サマリに加えて「期限前リマインド」(毎日1回)を将来追加しても約1,800通/月で枠内に収まります。
※ただし**ライトプランも追加メッセージの購入はできず、上限は5,000通/月**です。将来「タスク検出ごとの即時通知」を30名のグループに送ると月1万通を超え得るため(例: 5件/日 × 30名 × 30日 = 4,500通/月でほぼ上限)、即時通知を導入する場合は①宛先を少人数の専用グループにする、②スタンダードプラン(月15,000円・30,000通・追加購入可)へ変更する、のいずれかが必要です。

※今回の機能追加(会話の文脈解析・返信提案の生成)でGemini APIの呼び出しは増えますが、文脈を付与しても1件あたりのトークン量は小さく(flash-liteは100万トークンの文脈窓・入力$0.25/出力$1.50 per 1Mと安価)、引き続き無料〜数百円/月の見込みです。受領データの保存には**現在お使いのDropbox**を利用するため、本システムによる新たな費用は発生しません(保存容量は現在のご契約に依存し、店舗数が増えると有料プランが必要になる場合があります)。以上より**月額の合計は5,000円(税別)で据え置き**です。

※LINEは**2026年10月にメッセージ料金体系の改定(追加メッセージ部分)**が予定されています。現状の推奨(ライトプラン・約900〜1,800通/月)は5,000通の枠内で影響を受けませんが、将来「即時通知」等で枠を超える運用を検討する際は最新の料金をご確認ください。

## 9. 制約・注意点

1. **1対1トークは収集不可:** Botが見えるのはBotが参加しているグループ内のみ。グループ移行が完了するまで、既存の1対1トークは従来の人手運用が残ります(移行計画で段階的に解消)。
2. **Botのグループ参加許可設定:** LINE Developersコンソール > 対象チャネルの[Messaging API設定]タブで「グループトーク・複数人トークへの参加を許可する」をONにします(デフォルトはOFF。LINE Official Account Managerのアカウント設定からも変更可)。
3. **チャット機能との併用:** 2022年11月の仕様変更により、Webhook(Messaging API)と公式アカウントのチャット(手動返信)は併用可能です。本構成では二重応答を防ぐため、標準機能の「応答メッセージ」(自動あいさつ等)のみOFFにします。
4. **1グループに公式アカウントは1つまで:** お客様側で別のLINE公式アカウントがすでに参加しているグループには本Botを追加できません。その場合は新しいグループを作成して移行します。
5. **画像・ファイル・データの扱い(Dropbox自動保存):** 受領した画像・ファイル・データは**受信時に自動でDropboxへ保存**し、共有リンクをタスク行の「納品データ」列に記入します(要望③)。**LINEのコンテンツは保持期間が保証されず一定期間で自動削除される**ため、5分おきの分析バッチではなく**受信時に即時取得**する設計とします(取りこぼし防止)。本文の意味解析(分類・要約)は引き続きテキストを対象とします。
6. **AIの誤分類:** 100%の精度は出ないため、「迷ったらタスク化」+人がシートで「対象外」処理する設計で漏れを防ぎます。パイロット期間にプロンプトを調整します。
7. **個人情報の取り扱い:** メッセージ本文がGoogle(GAS/シート/Gemini)に送信され、受領した画像・ファイルはDropboxに保存されます。社内規程・お客様への説明(上記案内文)で担保してください。
8. **GAS固有の技術的注意点:**
   - LINE DevelopersのWebhook「検証」ボタンは、GASのWebアプリが302リダイレクトを返す仕様のため**失敗しますが正常**です(実際のメッセージ配信は問題なく処理されます。疎通確認は実メッセージで行います)。
   - GASはHTTPリクエストヘッダーを取得できないため、LINE公式推奨の署名検証(X-Line-Signature)が使えません。代替として、Webhook URLへの秘密トークン付与+リクエスト内の`destination`(BotのID)照合で不正リクエストを防ぎます。
   - スプレッドシートへの同時書き込み(Webhook受信と分析バッチの競合)はLockServiceで排他制御します。
   - 分析バッチのトリガーは5分おき程度とし、GAS無料枠(トリガー実行時間90分/日)内に収めます(1分おき等の高頻度はAI呼び出しの待ち時間で枠を超過するリスクがあります)。
   - **Dropbox保存はWebhook受信時に同期実行**します(コンテンツの失効防止のため分析バッチに回さない)。Dropboxのアクセストークンは4時間で失効するため、**長期有効なリフレッシュトークン**を保管し、`UrlFetchApp`で都度アクセストークンを再発行してアップロード(`content.dropboxapi.com/2/files/upload`)します。大容量ファイルはGASの実行時間・サイズ上限に留意します。
9. **Gemini無料枠の確認:** 無料枠の上限値(目安: 15リクエスト/分・1,000件/日)は変動するため、正確な値はGoogle AI Studioのレート制限画面で確認してください。
